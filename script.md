#### 8. LLM이란 무엇인가

**화면**
- **Large Language Model** — 방대한 텍스트로 학습한 대규모 신경망 — 언어를 생성·처리
- **핵심 동작** — 지금까지의 텍스트로 '다음에 올 토큰'을 확률로 예측
- **'이해'가 아니라 '패턴'** — 의미를 아는 게 아니라 통계적 패턴을 재현하는 것
- **예시** — '커피를 마시면 잠이 ___' → 다음에 올 토큰을 확률로 고른다

**대본/진행**

LLM은 인터넷 규모의 텍스트로 학습한 거대한 확률 모델입니다. 우리는 '이해한다'고 느끼지만, 내부에서는 '이 문맥 다음에 어떤 토큰이 올 확률이 높은가'를 계산할 뿐이에요. 비유하면, 수많은 문장을 읽은 사람이 빈칸 뒤에 올 말을 '아마 이거겠지' 하고 채우는 것과 같습니다. 의미를 진짜로 아는 게 아니라 통계적 패턴을 재현한다는 점 — 이게 뒤에 나올 환각의 뿌리입니다.

#### 9. 입력에서 출력까지

**화면**
- 토큰화(문장을 토큰(단어 조각)으로 분해) → 예측(다음 토큰의 확률 분포 계산) → 샘플링(확률에 따라 토큰 1개 선택) → 반복(붙이고 다시 예측 — 끝까지)

**대본/진행**

동작을 한 줄로 보면 이렇습니다. 먼저 문장을 토큰이라는 작은 조각으로 쪼개고, 각 시점에서 다음 토큰의 확률 분포를 계산해, 그중 하나를 골라 붙입니다. 그리고 이 과정을 끝날 때까지 반복하죠. 중요한 건 '한 번에 한 토큰씩'이라는 점입니다 — 그래서 긴 답일수록 앞에서의 작은 흔들림이 누적될 수 있어요. 이 자기회귀(autoregressive) 구조를 알면 temperature나 환각이 왜 생기는지 자연스럽게 이해됩니다.

- **길이·비용** — 영어 ~4글자≈1토큰 · 한글은 더 잘게 → 토큰 수가 비용·길이를 결정

#### 14. 사전학습 (Pre-training)

**화면**
- **방대한 텍스트로 자기지도 학습** — 웹·책·코드에서 '다음 토큰 맞히기'를 반복
- **무엇을 배우나** — 문법·세상 지식·추론 패턴·문체까지 광범위하게
- **한계의 씨앗** — 학습 시점 이후 정보는 모름 · 데이터 편향을 흡수
- **비유** — 거대한 빈칸 채우기를 수십억 번 — 그 과정에서 지식이 '딸려' 학습

**대본/진행**

모델은 두 단계로 만들어집니다. 첫 단계가 사전학습으로, 웹·책·코드 같은 방대한 텍스트에서 '다음 토큰 맞히기'를 끝없이 반복해요. 이 과정에서 문법뿐 아니라 세상 지식, 추론 패턴, 문체까지 폭넓게 흡수합니다. 단, 학습한 시점 이후의 정보는 모르고, 데이터에 있던 편향도 그대로 들어옵니다. 예를 들어 작년에 학습한 모델은 올해 새 논문을 모르죠 — 그래서 나중에 검색·RAG가 필요해집니다.

#### 15. 정렬 (Alignment)

**화면**
- **사전학습만으론 부족** — '유용하고·안전하게·지시를 따르도록' 추가로 다듬음
- **instruction tuning** — 지시–응답 예시로 '시키는 대로' 하도록 미세조정
- **인간 피드백 (RLHF 등)** — 사람이 선호하는 답을 강화 — 도움되고 해롭지 않게
- **사례** — InstructGPT(2022) — 정렬한 1.3B 답을 175B GPT-3보다 사람이 선호(~71%)

**대본/진행**

사전학습만 끝난 모델은 똑똑하지만 말을 잘 안 듣습니다. 그래서 두 번째 단계인 정렬에서, 사람이 원하는 방식으로 — 지시를 따르고, 안전하고, 도움이 되게 — 다듬습니다. 대표적으로 '지시-응답' 예시로 미세조정하는 instruction tuning, 그리고 사람이 선호한 답에 더 높은 점수를 주는 인간 피드백(RLHF 같은) 방식이 있어요. 우리가 ChatGPT나 Claude에서 느끼는 '말귀를 알아듣는' 느낌이 바로 이 정렬의 결과입니다.

#### 20. 추론(사고)하는 모델

**화면**
- **일반 생성 vs 추론** — 바로 답하지 않고, 단계적으로 '생각'한 뒤 답한다
- **언제 강한가** — 복잡한 문제 · 다단계 추론 · 코딩
- **트레이드오프** — 더 정확하지만 더 느리고 비싸다
- **사례** — DeepSeek-R1(2025·오픈)·OpenAI o1→o3 — AIME 78%→96.7%

**대본/진행**

최근에는 바로 답하지 않고, 속으로 단계를 밟아 '생각'한 뒤 답하는 추론 모델이 나왔습니다. 복잡한 수학, 다단계 논리, 코딩처럼 한 번에 답하기 어려운 문제에서 특히 강해요. 대신 생각하는 시간과 토큰을 더 쓰기 때문에 느리고 비쌉니다. 그래서 모든 작업에 추론 모델을 쓰는 게 아니라, 어려운 문제에만 골라 쓰는 판단이 필요합니다 — 뒤의 '모델 선택'과 연결됩니다.

#### 21. 임베딩 & 의미 검색

**화면**
- **임베딩이란** — 텍스트를 의미를 담은 숫자 벡터로 변환
- **의미 검색** — 글자가 달라도 '뜻이 비슷한' 문서를 찾음 (키워드 매칭의 한계 극복)
- **어디에 쓰나** — 방대한 문헌에서 관련 자료 검색 — 리서치 에이전트의 토대
- **사례** — Etsy·Spotify 검색/추천이 임베딩 기반

**대본/진행**

이제 외부 지식을 붙이는 첫 조각, 임베딩입니다. 임베딩은 텍스트를 '의미를 담은 숫자 벡터'로 바꾸는 기술이에요. 그러면 글자가 달라도 뜻이 가까운 문장끼리는 벡터도 가까워집니다. 예를 들어 '심장마비'와 '심근경색'은 글자는 다르지만 의미 검색에서는 비슷한 것으로 잡혀요. 키워드가 정확히 일치하지 않아도 관련 자료를 찾을 수 있다는 것 — 이게 방대한 문헌을 뒤지는 리서치 에이전트의 핵심 도구입니다.

#### 22. 검색 증강 생성 (RAG)

**화면**
- 질문(사용자 질문) → 검색(관련 문서 찾기(임베딩)) → 주입(찾은 근거를 프롬프트에) → 생성(근거 기반으로 답)

**대본/진행**

임베딩으로 찾은 자료를 LLM에 '먹여서' 답하게 하는 게 RAG입니다. 흐름은 간단해요: 질문이 들어오면, 관련 문서를 검색하고, 그 근거를 프롬프트에 넣은 다음, 모델이 그 근거를 바탕으로 답합니다. 이렇게 하면 두 가지가 좋아집니다 — 모델이 모르던 최신·전문 정보를 답할 수 있고, 근거가 있으니 환각이 줄어듭니다. 이번 주 팀 프로젝트가 바로 이 RAG 패턴의 미니 버전이라고 보면 됩니다.

#### 23. 질문 — 무엇을 묻나

**화면**
- **입력** — 사용자의 질문이 출발점
- **필요시 분해** — 큰 질문이면 검색 가능한 작은 질의로
- **목표** — '무슨 근거를 찾아야 하나'를 정한다

**대본/진행**

*실습·진행 슬라이드 — 강사가 직접 진행 (대본 생략)*

#### 24. 검색 — 관련 문서 찾기

**화면**
- **방법** — 질문을 임베딩해 '의미가 가까운' 문서를 검색
- **대상** — 문서 폴더·DB·웹 등 (이번 프로젝트는 로컬 corpus)
- **결과** — 관련 문단 몇 개 + 출처

**대본/진행**

*실습·진행 슬라이드 — 강사가 직접 진행 (대본 생략)*

#### 25. 주입 — 근거를 프롬프트에

**화면**
- **무엇** — 찾은 문단을 프롬프트 안에 같이 넣어줌
- **효과** — 모델이 '기억'이 아니라 '눈앞의 근거'로 답하게
- **주의** — 너무 많이 넣으면 컨텍스트 한계·비용↑

**대본/진행**

*실습·진행 슬라이드 — 강사가 직접 진행 (대본 생략)*

#### 26. 생성 — 근거 기반 답

**화면**
- **무엇** — 주입된 근거를 바탕으로 답 생성
- **핵심** — 출처를 함께 제시 → 검증 가능
- **효과** — 환각↓ · 최신·전문 정보 반영↑
- **사례** — OpenEvidence — 동료심사 문헌(NEJM·JAMA) 기반 의사용 Q&A (약학 RAG)

**대본/진행**

*실습·진행 슬라이드 — 강사가 직접 진행 (대본 생략)*

#### 27. 프롬프트가 동작을 바꾼다

**화면**
- **프롬프트 = 모델에 주는 입력 전체** — 지시 + 맥락 + 예시 + 데이터
- **좋은 프롬프트** — 명확한 목표 · 충분한 맥락 · 원하는 형식 명시
- **나쁜 프롬프트** — 모호·맥락 부족 → 엉뚱하거나 일반적인 답
- **비교 예시** — '코딩 도와줘' vs 'CSV 항목별 합계 코드 만들고 실행·확인해줘'

**대본/진행**

같은 모델이라도 프롬프트를 어떻게 주느냐에 따라 결과가 완전히 달라집니다. 프롬프트는 단순한 질문이 아니라 '지시 + 맥락 + 예시 + 데이터'를 합친 입력 전체예요. 좋은 프롬프트는 목표가 분명하고, 충분한 맥락을 주고, 원하는 출력 형식을 명시합니다. 반대로 모호하게 던지면 모델은 가장 무난하고 일반적인 답을 내놓죠. '코딩 도와줘'와 '이 CSV를 항목별로 합산하는 파이썬 코드를 만들고 실행해서 확인해줘'의 차이를 떠올리면 됩니다.

#### 28. 자주 쓰는 패턴 — 3가지

**화면**
- **역할·맥락 부여** — 누가·무엇을·왜
- **예시 제공 (few-shot)** — 원하는 입출력 예를 보여주기
- **단계적 사고 (CoT)** — 단계별로 생각하게
- **사례** — GPT-3(2020) — 예시 몇 개(few-shot)만으로 번역·Q&A 수행

**대본/진행**

*실습·진행 슬라이드 — 강사가 직접 진행 (대본 생략)*

#### 29. 역할·맥락 부여

**화면**
- **방법** — '너는 약학 전문가야' + 상황·목적·대상 독자
- **효과** — 답의 톤·깊이·용어 수준이 맞춰짐
- **예** — '비전공자에게 설명하듯' vs '전문가용으로'

**대본/진행**

*실습·진행 슬라이드 — 강사가 직접 진행 (대본 생략)*

#### 30. 예시 제공 (few-shot)

**화면**
- **방법** — 원하는 입력→출력 예를 2~3개 보여준 뒤 본 작업 요청
- **효과** — 형식·스타일이 안정 (말로 설명하기 어려운 형식에 특히)
- **예** — 표 형식 예시 2개 → 새 데이터도 같은 표로

**대본/진행**

*실습·진행 슬라이드 — 강사가 직접 진행 (대본 생략)*

#### 31. 단계적 사고 (CoT)

**화면**
- **방법** — '단계별로 생각해줘' / '근거를 먼저 정리한 뒤 답해줘'
- **효과** — 복잡한 추론·계산의 정확도↑ (생각을 펼치게)
- **주의** — 간단한 작업엔 굳이 — 길어지고 느려짐

**대본/진행**

*실습·진행 슬라이드 — 강사가 직접 진행 (대본 생략)*

#### 32. 같은 모델, 다른 출력

**화면**
- **temperature** — 낮으면 결정적·일관, 높으면 다양·창의(불안정)
- **확률적 생성** — 같은 질문도 매번 다른 답이 나올 수 있다
- **재현성에의 함의** — 리서치엔 낮은 temperature가 유리 — 결과 안정

**대본/진행**

같은 질문을 줬는데 답이 매번 다른 경험, 해보셨을 겁니다. 이건 버그가 아니라 확률적 생성의 본질이에요. temperature라는 값이 이 다양성을 조절하는데, 낮으면 매번 비슷하고 안정적인 답을, 높으면 다양하지만 불안정한 답을 줍니다. 브레인스토밍에는 높게, 사실 기반 리서치에는 낮게 두는 게 좋아요. 특히 리서치에서는 재현성이 중요하니 temperature를 낮춰 결과를 안정시키는 걸 권합니다.

#### 33. 컨텍스트 윈도우

**화면**
- **한 번에 볼 수 있는 토큰 한계** — 프롬프트+대화+자료가 이 한도 안에 들어가야
- **길어지면** — 비용↑·속도↓·중간 내용 놓침(lost in the middle)
- **에이전트와의 관련** — 외부 메모리·검색·도구로 한계를 보완

**대본/진행**

모델이 '한 번에 볼 수 있는 정보의 양'에는 한계가 있는데, 이걸 컨텍스트 윈도우라고 합니다. 프롬프트, 지금까지의 대화, 첨부한 자료가 모두 이 한도 안에 들어가야 해요. 너무 길어지면 비용과 속도가 나빠지고, 가운데 있는 내용을 놓치는 현상(lost in the middle)도 생깁니다. 그래서 에이전트는 모든 걸 컨텍스트에 욱여넣는 대신, 외부 메모리와 검색으로 '필요할 때 필요한 것만' 가져오도록 설계합니다.

#### 34. LLM의 능력과 한계

**화면**
- 잘하는 것: 요약·번역·분류 · 코드 생성·설명 · 텍스트 추출·재구성 · 브레인스토밍
- 주의할 것: 환각 — 없는 사실 지어냄 · 최신·실시간 정보 · 정확한 계산·셈 · 일관된 장기 추론

**대본/진행**

여기서 균형 잡힌 시각이 중요합니다. LLM은 요약·번역·분류, 코드 생성, 텍스트 추출·재구성, 브레인스토밍에 아주 강해요. 반대로 없는 사실을 지어내는 환각, 최신·실시간 정보, 정확한 계산이나 셈, 아주 긴 추론의 일관성에는 약합니다. 핵심 메시지는 'LLM을 만능으로 믿지 말고, 강점은 활용하고 약점은 도구로 보완하라'는 거예요. 다음 슬라이드의 환각이 그 약점의 대표 선수입니다.

#### 35. 환각(hallucination) 바로 알기

**화면**
- **그럴듯하지만 틀린 출력** — 출처·인용·수치를 자신있게 지어내기도
- **왜 생기나** — '그럴듯한 다음 토큰'을 생성할 뿐 사실 검증을 안 함
- **대응** — 출처 확인·검색 연동(RAG)·교차검증 → 리서치 에이전트의 존재 이유
- **실제 사례** — 변호사가 ChatGPT 가짜 판례 인용→벌금(Mata v. Avianca, 2023) · 에어캐나다 챗봇이 없는 환불정책 지어내 배상(2024)

**대본/진행**

환각은 모델이 '그럴듯하지만 틀린' 내용을, 그것도 자신 있게 만들어내는 현상입니다. 가짜 출처나 존재하지 않는 인용, 틀린 수치를 진짜처럼 제시하기도 해요. 왜 그럴까요? 모델은 사실을 검증하는 게 아니라 '그럴듯한 다음 토큰'을 생성할 뿐이기 때문입니다. 그래서 대응은 명확합니다 — 출처를 확인하고, 검색(RAG)으로 근거를 대고, 여러 자료로 교차검증하는 것. 바로 이게 우리가 리서치 에이전트를 만드는 이유입니다.

#### 36. 어떤 모델을 쓸까

**화면**
- **크기 ↔ 성능** — 큰 모델 = 똑똑하지만 느리고 비쌈
- **작업에 맞게** — 간단한 작업엔 작고 빠른 모델, 어려운 작업엔 크고/추론 모델
- **Claude 예** — Opus(최상위) · Sonnet(균형) · Haiku(빠름·저비용)

**대본/진행**

마지막 실용 팁, 모델 선택입니다. 큰 모델일수록 똑똑하지만 느리고 비싸요 — 성능과 비용은 대체로 맞바꿈 관계입니다. 그래서 간단한 작업에는 작고 빠른 모델을, 어렵고 중요한 작업에는 크거나 추론하는 모델을 쓰는 게 합리적입니다. Claude를 예로 들면 Opus는 최상위, Sonnet은 균형, Haiku는 빠른 처리에 적합하죠(구체 모델명은 시점에 따라 바뀌니 확인). 한 작업 안에서도 단계별로 다른 모델을 섞어 쓰는 경우도 많습니다.

#### 37. LLM → 에이전트로 가는 길

**화면**
- **LLM 단독의 한계** — 도구 없음 · 기억 없음 · 한 번의 응답으로 끝
- **RAG·검색을 루프로 = 에이전트** — 찾고 → 검증하고 → 종합 — 리서치 에이전트로
- **다음: Day 2** — 에이전트의 정의와 Claude Code 실습

**대본/진행**

정리하면, LLM은 강력하지만 혼자서는 도구도 기억도 없고 한 번 답하면 끝입니다. 여기에 '도구를 쓰고, 결과를 보고, 다시 시도하는 반복'을 붙인 게 바로 에이전트예요. 특히 오늘 배운 검색·RAG를 루프로 돌리면 — 찾고, 검증하고, 종합하는 — 리서치 에이전트가 됩니다. 즉 오늘의 임베딩과 RAG가 이번 주 프로젝트의 토대입니다. 내일은 이 에이전트의 정의를 자세히 보고, 실습 도구인 Claude Code를 직접 설치합니다.

## Day 2 (화) — AI 에이전트 원리 + 클로드 코드

### 오전 특강 · AI 에이전트 원리와 이해 + AI 에이전트 기반 리서치  (90분 · 10:00–11:30)

*세션 목표: 에이전트의 정의·구조·자율성과, 리서치 에이전트의 전체 구조를 이해한다.*

#### 1. 이 세션에서 다룰 것

**화면**
- **에이전트의 정의와 구조** — LLM + 도구(tools) + 반복(loop)
- **바이브코딩 작업 방식** — 자연어로 만들고, 검증하며 반복
- **리서치 에이전트의 큰 틀** — 이번 주 팀 프로젝트의 형태

**대본/진행**

어제는 LLM 자체를 봤다면, 오늘은 그 LLM을 '행동하게' 만드는 에이전트를 배웁니다. 핵심 공식은 간단해요 — 에이전트 = LLM + 도구 + 반복. 그리고 코드를 말로 짜는 바이브코딩, 마지막으로 이번 주 프로젝트의 형태인 리서치 에이전트의 큰 틀을 잡습니다. 오후에는 바로 이 에이전트 도구인 Claude Code를 직접 설치합니다.

#### 2. AI 에이전트란?

**화면**
- **정의** — LLM(두뇌) + 도구(tools) + 반복 루프(loop)로 목표를 수행
- **LLM과의 차이** — 답만 내는 게 아니라 '행동'하고 결과를 보고 다음 행동을 결정
- **예** — 검색 → 읽기 → 코드 실행 → 수정을 스스로 반복
- **사례** — Claude Code(코드 읽기→수정→테스트→반복) · Claude Computer use(화면 보고 클릭, 2024)

**대본/진행**

LLM은 질문에 답만 하지만, 에이전트는 한발 더 나아가 '행동'합니다. 두뇌인 LLM에 검색·실행 같은 도구를 쥐여주고, 결과를 보고 다음 행동을 스스로 정하는 반복 루프를 붙인 거예요. 예를 들어 '버그 고쳐줘'라고 하면 코드를 읽고 → 수정하고 → 실행해보고 → 안 되면 다시 고치는 과정을 알아서 돕니다. 즉 한 번의 응답이 아니라, 목표를 향한 여러 단계의 작업이 에이전트의 본질입니다.

#### 3. LLM vs 에이전트

**화면**
- LLM 단독: 입력→출력 한 번 · 도구 없음 · 기억 없음 · 사용자가 매번 지시
- 에이전트: 목표를 주면 · 도구를 골라 실행 · 상태·기억 유지 · 스스로 반복·판단

**대본/진행**

이 둘의 차이를 분명히 해둡시다. LLM 단독은 텍스트를 받아 텍스트를 내놓는 일회성 응답이고, 도구도 기억도 없어 매번 사람이 지시해야 합니다. 반면 에이전트는 목표를 주면 스스로 도구를 골라 쓰고, 상태와 기억을 유지하며 될 때까지 반복합니다. 표의 왼쪽이 '똑똑한 자동완성'이라면, 오른쪽은 '일을 맡길 수 있는 조수'라고 보면 됩니다.

#### 4. 에이전트의 동작 사이클

**화면**
- 관찰(상황·이전 결과 파악) → 사고(다음에 무엇을 할지 결정) → 행동(도구 호출(검색·실행)) → 관찰(결과 확인 → 반복)

**대본/진행**

에이전트가 어떻게 일하는지 한 사이클로 보면 — 관찰하고, 생각하고, 행동하고, 다시 관찰합니다. 현재 상황을 보고, 다음에 뭘 할지 정하고, 도구를 실행한 뒤, 그 결과를 다시 관찰해 또 판단하죠. 이 루프를 목표가 달성될 때까지 반복합니다. 사람이 일하는 방식과 똑같아요 — 해보고, 결과 보고, 다음 수를 두는 겁니다.

**화면**

- **예** — 코드 에이전트가 테스트 실행 결과(통과/실패)를 읽음

#### 6. 사고 (Think)

**화면**
- **무엇** — 관찰을 바탕으로 다음에 무엇을 할지 결정
- **예** — '테스트가 실패했으니 이 함수를 고쳐야겠다'

#### 7. 행동 (Act)

**화면**

#### 9. 도구(tools) — 에이전트의 손과 발

**화면**
- **도구란** — 에이전트가 호출하는 외부 기능 — 웹검색·파일·코드실행·API
- **왜 필요한가** — LLM의 한계(최신성·계산·실행)를 실제 행동으로 보완
- **동작 방식** — LLM이 '어떤 도구를 어떤 인자로' 호출할지 결정 → 결과를 다시 입력
- **사례** — ChatGPT Search — LLM이 스스로 검색 쿼리를 만들어 웹 검색 후 출처 인용(2024)

**대본/진행**

에이전트가 행동할 수 있는 건 '도구' 덕분입니다. 웹 검색, 코드 실행, 파일 읽기·쓰기, 외부 API 호출 같은 외부 기능들이죠. 왜 필요하냐면, 어제 본 LLM의 한계 — 최신 정보를 모르고, 계산을 틀리고, 실제로 실행하지 못하는 — 를 도구가 메워주기 때문입니다. 동작 방식은, LLM이 '어떤 도구를 어떤 값으로 부를지' 정하면 시스템이 실행하고 그 결과를 다시 LLM에 돌려주는 식입니다.

#### 10. 메모리와 컨텍스트

**화면**
- **단기 (컨텍스트)** — 현재 작업 맥락 — 컨텍스트 윈도우 안에 유지
- **장기 (외부 메모리)** — 파일·DB·벡터스토어에 저장해 필요할 때 검색
- **왜 중요한가** — 긴 작업의 일관성 유지 · 맥락 한계 극복

**대본/진행**

긴 작업을 하려면 기억이 필요합니다. 단기 기억은 지금 대화·작업 맥락으로, 어제 배운 컨텍스트 윈도우 안에 들어갑니다. 장기 기억은 그 한계를 넘기 위해 파일이나 데이터베이스에 저장해 두고 필요할 때 검색해 오는 거예요. 이게 있어야 에이전트가 긴 작업에서도 앞에서 한 일을 잊지 않고 일관되게 진행합니다.

#### 11. 자율성의 정도

**화면**
- **보조형 (copilot)** — 사람이 매 단계 확인·승인
- **반자율** — 일상 작업은 자동, 중요한 결정만 사람이 확인
- **자율형** — 목표만 주면 스스로 — 강력하지만 검증·안전장치 필수
- **사례 스펙트럼** — GitHub Copilot(보조) → Copilot Workspace(반자율) → Devin(완전자율)

**대본/진행**

에이전트라고 다 똑같이 자율적인 건 아닙니다. 사람이 매 단계 확인하는 보조형, 일상은 자동이고 중요한 결정만 확인받는 반자율, 목표만 주면 끝까지 스스로 하는 자율형으로 스펙트럼이 있어요. 자율성이 높을수록 강력하지만 그만큼 검증과 안전장치가 중요해집니다. 우리 실습에서는 사람이 확인하며 진행하는 쪽에 가깝게 씁니다.

#### 12. 바이브코딩 (Vibe Coding)

**화면**
- **정의** — 코드를 직접 타이핑하지 않고, 자연어로 의도를 전달해 만든다
- **사람의 일** — 무엇을·왜를 명확히 + 결과를 검증하고 방향 제시
- **핵심** — 빠른 반복 + 반드시 검증 (출력을 그대로 신뢰 금물)
- **사례** — 용어 기원 Karpathy(2025.2) · Fly.Pieter.com — 1인이 프롬프트로 ~3시간 만에 비행게임

**대본/진행**

이제 작업 방식이 어떻게 바뀌는지 봅시다. 바이브코딩은 코드를 한 줄씩 직접 타이핑하는 대신, '무엇을 원하는지'를 자연어로 말하면 에이전트가 구현해 주는 방식입니다. 그래서 여러분의 역할은 코드 작성이 아니라 의도를 분명히 전달하고 결과를 검증하는 것으로 바뀝니다. 핵심은 '빠르게 반복하되 반드시 검증한다' — 에이전트가 만든 걸 그냥 믿으면 안 됩니다.

#### 13. 바이브코딩 사이클

**화면**
- 의도 전달(원하는 것을 구체적으로) → 생성(에이전트가 구현) → 실행(실제로 돌려 확인) → 검증·수정(결과 보고 다시 지시)

**대본/진행**

바이브코딩도 사이클입니다 — 원하는 걸 구체적으로 전달하고, 에이전트가 만들고, 실제로 돌려서 확인하고, 결과를 보고 다시 지시합니다. 비결은 한 번에 다 만들려 하지 말고 작은 단위로 쪼개는 거예요. 작게 만들고 확인하면 어디서 틀렸는지 금방 보이고 방향도 쉽게 잡힙니다. 오후 실습에서 이 리듬을 몸으로 익히게 됩니다.

#### 1. 왜 기획이 먼저인가

**화면**
- **방향 없는 구현은 헤맨다** — 기획이 절반
- **범위를 좁혀야 끝낼 수 있다** — 한 세션 안에 결과
- **'완성'의 정의를 먼저** — 산출물 합의

**대본/진행**

바로 만들고 싶겠지만, 기획 없이 시작하면 십중팔구 헤맵니다. 방향이 없으면 에이전트에게 뭘 시킬지 모호하고, 범위가 안 정해지면 끝이 안 나요. '무엇이 완성인지'를 먼저 합의하는 게 절반입니다 — 오늘은 한 세션 안에 끝낼 수 있는 작은 한 가지를 잡습니다.

#### 2. 좋은 미니 프로젝트 고르기

**화면**
- **작고 명확하게** — 짧은 시간에 결과가 보이는 범위
- **실생활 문제에서 출발** — 예: 가계부 요약 · 일정 정리 · 자료 검색 도우미
- **도구로 실현 가능한가** — 현실성 점검

**대본/진행**

조건은 셋이에요. 작고 명확할 것(짧은 시간에 결과가 보여야 함), 실생활 불편에서 출발할 것(가계부·일정·자료검색 같은), 지금 도구로 실현 가능할 것. 멋진 아이디어보다 '오늘 끝낼 수 있는' 아이디어가 좋은 주제입니다.

#### 3. 문제를 한 문장으로

**화면**
- **누구의 어떤 불편인가** — 문제 진술
- **왜 지금 풀 가치가 있나** — 지금 불편이 크거나 자주 생기나
- **성공하면 무엇이 달라지나** — 전·후 차이를 한 줄로

**대본/진행**

주제를 정했으면 문제를 딱 한 문장으로 — '누구의 어떤 불편을 해결한다'. 이게 막연하면 만드는 내내 흔들려요. 왜 지금 풀 가치가 있는지, 성공하면 뭐가 달라지는지를 한 줄씩 더하면 방향이 또렷해집니다.

#### 4. 기획 시 정할 것

**화면**
- 문제 / 사용자: 해결할 문제 · 누가 쓰나 · 성공 기준 · 사용 맥락
- 범위 / 산출물: 할 것 / 안 할 것 · 필요한 도구 · 결과 형태 · 마감

**대본/진행**

기획서는 두 축입니다 — 왼쪽은 문제·사용자(무엇을·누가·성공 기준), 오른쪽은 범위·산출물(할 것/안 할 것·도구·결과·마감). 특히 '안 할 것'을 분명히 적어 범위를 좁히세요. 템플릿은 examples 폴더에 있으니 채우면 됩니다.

### 오후 특강 2 · 리서치 프로세스(기존·agentic) + 쟁점 토론 + 팀·주제 설정  (90분 · 14:30–16:00)

*세션 목표: 기존 연구 프로세스와 그 agentic 버전·핵심 쟁점을 함께 본 뒤, 팀과 리서치 주제를 정한다.*

#### 1. 기존 연구 프로세스 (사람)

**화면**
- 분야 파악(전체를 훑기) → 관심 좁히기(세부 주제) → 문헌 선별·검증(선별 → 비판적 읽기) → 가설 수립(gap·보완) → 우선순위·검증(높은 것부터)

**대본/진행**

에이전트로 자동화하기 전에, 사람이 연구를 어떻게 하는지부터 봅시다. 보통 분야 전체를 훑고, 세부 관심을 좁히고, 논문을 선별·검증하고, 개선 가설을 세우고, 우선순위를 매겨 검증하죠. 그런데 현실은 이 그림처럼 한 방향으로 흐르지 않습니다 — 가설을 세우다 보면 어떤 논문이 중요한지가 바뀌고, 검증하다 새 공백이 보여요. 즉 직선이 아니라 반복 루프입니다. 또 '재현'은 비용이 커서, 실제로는 논문 전체를 다시 돌리기보다 주장과 근거를 비판적으로 점검하는 '검증'에 가깝습니다.

#### 2. 각 단계를 agent로 — 5개 질문

**화면**
- **① 분야 파악 / ② 논문 선별** — 맵·taxonomy 만들기 / funnel 랭킹(다양성 포함)
- **③ 논문 학습 / ④ 가설 생성** — 구조화 추출·모순 그래프 / gap·조합 기반
- **⑤ 가설 우선순위** — 스코어링·랭킹 (impact × plausibility ÷ 비용)

**대본/진행**

그럼 이 각 단계를 에이전트로 어떻게 옮길까, 라는 다섯 가지 질문이 자연스럽게 나옵니다. 분야 파악은 다 읽는 게 아니라 리뷰 논문에서 출발해 '지도(맵)'를 만드는 것이고, 논문 선별은 넓게 모은 뒤 깔때기처럼 좁히되 모순·이상치도 일부러 남깁니다. 논문 학습은 '읽기'가 아니라 동일한 틀로 구조화 추출해 모순 그래프를 만드는 것이고요. 가설 도출은 공백·모순·조합에서, 우선순위는 영향력·타당성·비용을 곱하고 나눠 정합니다. 이 다섯 단계가 곧 리서치 에이전트가 실제로 다루는 부분입니다.

#### 3. 가설: '다수 생성' vs '선별'

**화면**
- 다수 생성 (breadth): 가용 데이터/도구로 제약 · 싼 테스트로 거르기 · 다양성↑ · LLM 판단 의존↓
- 사전 선별 (selection): 비싼 테스트 전에 추리기 · LLM 판단에 의존 · 위양성↓ · 놓칠 위험

**대본/진행**

여기서 오늘의 핵심 쟁점입니다. 좋은 가설을 얻는 두 가지 길이 있어요. 하나는 비싼 테스트 전에 '될 만한 것'을 똑똑하게 골라내는 선별 전략, 다른 하나는 일단 많이·다양하게 만들어 놓고 싼 테스트로 거르는 다수 생성 전략입니다. 학생들에게 먼저 물어보세요 — 여러분 주제라면 어디에 무게를 두겠냐고. 한쪽은 LLM의 판단력에 의존하고, 다른 쪽은 데이터·도구로 가설을 제약한 뒤 실측으로 거릅니다. 정답을 바로 주지 말고 잠깐 토론을 유도하세요.

#### 4. 테스트가 싸지면 무게중심이 이동

**화면**
- **agent의 강점/약점** — 생성·싼 테스트 throughput ↑ / 새 가설 품질 판단은 약함
- **권고** — 제약된 다수 생성 → 싼 경험적 필터 → 비싼 검증은 소수만
- **주의** — 다중검정 위양성 · 싼 테스트는 proxy (특히 wet-lab/임상)

**대본/진행**

제 권고는 이렇습니다. 에이전트 덕분에 테스트 비용이 확 싸지는 상황이라면, 무게중심은 '제약된 다수 생성 + 싼 경험적 필터' 쪽으로 기울어야 합니다. 이유는, 새 가설의 품질을 미리 판단하는 건 LLM이 가장 약한 부분이거든요 — 차라리 현실에 근거한 싼 테스트로 거르는 게 더 믿을 만합니다. 단 두 가지를 조심해야 해요. 많이 테스트하면 우연히 '성공처럼 보이는 것'이 늘어나는 다중검정 문제, 그리고 싼 테스트는 보통 진짜 결과의 대리물이라는 점입니다. 특히 신약·임상은 최종 검증이 에이전트로도 싸지지 않으니, '싼 인실리코로 넓게 → 좁히고 → 비싼 실측'의 단계 전략으로 가야 합니다.

#### 5. 프로젝트 형식 + 3세션 스코프

**화면**
- **무엇** — 자기 관심 분야에서 'rationale 갖춘 새 가설'을 발견·제안하는 에이전트
- **스코프** — corpus 기반 · 자료파악→gap발견→가설생성→평가·제안 · Day 4 안에 완주
- **산출물** — rationale 갖춘 새 연구 가설 제안서 (Q&A 보고서 아님)

**대본/진행**

이제 프로젝트를 안내합니다. 각 팀은 자기 관심 주제로 '풀 리서치 에이전트'를 만듭니다 — 한 단계만 떼는 게 아니라 전체 파이프라인을요. 단, 내일 세 세션 안에 완주할 수 있도록 스코프를 좁힙니다. 핵심 제한은 'corpus 기반' — 오픈 웹을 긁는 대신, 제공하거나 직접 모은 문서 폴더에서 검색합니다. 산출물은 핵심 질문에 답하는 1~2장짜리 인용 보고서예요. 욕심내지 말고 작게 완주하는 게 목표라는 걸 강조하세요.

#### 6. 가설 생성 파이프라인 (만들 것)

**화면**
- 주제 설정(관심 분야·발견 목표) → 자료 파악(로컬 corpus 정리) → gap 발견(모순·공백·미검증) → 가설 생성(gap·조합 기반 다수) → 평가·제안(rationale·우선순위)

**대본/진행**

여러분이 내일 만들 흐름이 이겁니다 — 관심 분야의 자료를 파악하고, 그 안에서 모순·공백(gap)을 찾아내고, 그 gap에서 새 가설을 여러 개 만들고, 근거·새로움·타당성으로 평가해서, rationale 갖춘 가설을 제안하는 것. 핵심은 '질문에 답하기'가 아니라 'gap에서 새 가설을 만들기'라는 점이에요. 처음부터 다 만들려 하지 말고, 자료 파악 → gap 하나 → 가설 하나의 최소 버전을 먼저 돌린 뒤 다수 생성과 평가를 붙이세요. 이 그림을 머릿속에 넣고 기획서를 쓰면 한결 수월합니다.

#### 7. 팀 구성 & 분야·기획

**화면**
- 단계: 팀 편성·역할 분담 → 관심 분야 + 발견 목표(무엇을 제안하고 싶나) → 자료(corpus) 확보 계획 → 팀 기획서 작성
- 산출물: 팀 기획서 (분야 · 발견 목표 · corpus · 산출물=가설 제안 · 스코프) 예: 신약/바이오 리서치 등 템플릿: examples/research-agent/research-plan-template.md

**대본/진행**

[실습 안내] 이제 팀을 짜고 기획서를 씁니다. 먼저 팀과 역할을 정하고(자료 정리 / gap 발견 / 가설·평가 등), 관심 분야와 '무엇을 발견·제안하고 싶은지' 발견 목표를 한 문장으로 정하세요. 그다음 어떤 문서에서 gap을 찾을지 corpus 확보 계획을 적습니다 — 직접 모아도 되고 제공 자료를 써도 됩니다. 마지막으로 분야·발견 목표·corpus·산출물(가설 제안)·스코프를 한 장 기획서에 정리하세요. 다 쓰면 팀별로 한 줄씩 공유합니다. 내일 아침에는 이 분야를 짧게 발표하고 제가 직접 피드백을 드릴 테니, 오늘은 방향을 분명히 잡는 데 집중하세요.

### 오전 특강 · 팀별 리서치 AI Agent — 기획 및 논문 수집  (90분 · 10:00–11:30)

*세션 목표: 팀 발표·강사 피드백으로 방향을 확정하고, Claude Code로 PubMed에서 논문 corpus를 수집한다.*

#### 1. 팀별 발표 → 강사 피드백

**화면**
- **팀별 짧은 발표 (~2분)** — 분야 · 발견 목표 · 왜 가치 있나
- **강사 직접 피드백** — 스코프 적정? · 발견 목표 명확? · 어떤 논문을 모을까?
- **반영** — 피드백대로 조정 후 수집 시작

**대본/진행**

*실습·진행 슬라이드 — 강사가 직접 진행 (대본 생략)*

#### 2. 오늘 만들 것 — 가설 생성 파이프라인

**화면**
- 논문 수집(PubMed → corpus) → 자료 파악(구조화 정리) → gap 발견(모순·공백) → 가설 생성(다수) → 평가·제안(rationale·우선순위)

**대본/진행**

발표와 피드백으로 방향을 잡았으니, 이제 오늘 전체 흐름을 보겠습니다. 다섯 단계예요 — 논문을 수집하고, 그 자료를 파악하고, 거기서 gap을 찾고, gap에서 가설을 만들고, 평가해서 제안합니다. 오늘 하루를 한 세션에 한 단계씩 나눠 합니다. 오전은 재료가 되는 논문 수집, 오후1은 자료 파악과 gap, 오후2는 가설 생성과 평가. 한 단계씩 끝낼 때마다 결과를 같이 점검하고 넘어갈게요.

#### 3. PubMed에서 논문 모으기

**화면**
- **어디서** — PubMed — 의학·약학 논문 DB (무료 E-utilities API, 키 불필요)
- **어떻게** — Claude Code에 '검색어로 초록·메타데이터를 papers/ 폴더에 저장' 요청 (바이브코딩)
- **무엇을** — 제목·저자·연도·초록·PMID — 약 20~30편

**대본/진행**

오전의 핵심은 '진짜 재료'를 모으는 겁니다. 약학·의학 논문은 PubMed에서 찾는 게 자연스러워요 — 무료 API(E-utilities)가 있고 키도 필요 없습니다. 방법은 바이브코딩 그대로예요. Claude Code에게 '이 검색어로 PubMed에서 초록과 메타데이터를 받아 papers 폴더에 저장하는 스크립트를 만들고 실행해줘'라고 시키면 됩니다. 모으는 항목은 제목·저자·연도·초록·PMID이고, 20~30편 정도면 충분합니다. 검색어는 관심 키워드에 리뷰 필터나 최근 몇 년 조건을 더하면 품질이 올라갑니다.

#### 4. 논문 수집 실습 (Claude Code · PubMed)

**화면**
- 단계: 검색어 정의 (분야→키워드) → 수집 스크립트 작성·요청 (PubMed API) → 실행 → papers/ 에 ~20–30편 저장 → 샘플 2~3편 링크로 실제 논문 확인
- 산출물: papers/ 폴더 = 팀 corpus (초록·메타데이터) 참고: examples/research-agent/sources 형식

**대본/진행**

*실습·진행 슬라이드 — 강사가 직접 진행 (대본 생략)*

#### 5. 수집할 때 꼭 확인

**화면**
- **실제 논문인가** — ★ Claude가 지어낸 가짜 논문 주의 — PMID·링크로 확인
- **관련성 있나** — 검색어가 너무 넓/좁지 않은지 — 필요시 조정·재수집
- **규모 적당한가** — 20~30편 — gap 찾기엔 충분, 다루기엔 부담 적음

**대본/진행**

여기서 가장 중요한 주의 하나 — 진짜 논문인지 확인하세요. Claude에게 그냥 '논문 찾아줘' 하면 그럴듯한 가짜 논문을 지어낼 수 있습니다. 그래서 실제 API로 받아야 하고, 받은 것 중 두세 개는 PMID나 링크로 직접 열어 확인하세요. 그다음 관련성 — 검색어가 너무 넓거나 좁으면 조정해서 다시 모으고요. 규모는 20~30편이 적당합니다. 이 corpus 품질이 이후 가설 품질을 그대로 좌우하니, 오전에 제대로 모으는 게 중요합니다.

### 오후 특강 1 · 팀별 리서치 AI Agent — 자료 파악 & gap 발견  (80분 · 13:00–14:20)

*세션 목표: 수집한 논문을 같은 틀로 정리해 지도를 만들고, 모순·공백(gap)을 찾는다.*

#### 1. 오늘 오후1 — 자료 파악 → gap

**화면**
- **목표** — 수집한 논문에서 '빈틈(gap)'을 찾는다 — 가설의 원천
- **방법** — 먼저 같은 틀로 정리(파악) → 그래야 비교가 되고 gap이 보임
- **산출** — gap 목록 (각 gap에 근거 논문)

**대본/진행**

오후엔 모은 논문에서 '빈틈'을 찾습니다. 빈틈, 즉 gap이 바로 새 가설의 원천이거든요. 그런데 바로 gap을 찾으려 하면 안 보여요 — 먼저 논문들을 같은 틀로 정리해야 비교가 되고, 그래야 어긋나는 지점이 드러납니다. 그래서 순서가 '자료 파악 먼저, 그다음 gap'입니다. 이 세션 산출물은 gap 목록이고, 각 gap에는 어느 논문에서 나왔는지 근거를 답니다.

#### 2. 논문을 같은 틀로 정리

**화면**
- **왜** — '읽기'가 아니라 정규화 — 비교 가능해야 gap이 드러남
- **무엇을 뽑나** — 각 논문: 핵심 주장 · 방법 · 근거 · 한계 · 결론
- **결과** — 합의된 것 vs 논쟁 중인 것 지도
- **예시** — 논문마다 동일 표로 → 주장이 엇갈리는 행이 한눈에

**대본/진행**

자료 파악은 '읽기'가 아니라 '정규화'입니다. 논문마다 같은 항목을 뽑아요 — 핵심 주장, 방법, 근거, 한계, 결론. 이렇게 같은 표에 넣으면 어느 논문이 무슨 주장을 하는지 한눈에 비교됩니다. 그러면 자연스럽게 '합의된 것'과 '논쟁 중인 것'이 갈리죠. 주장이 엇갈리는 행, 그게 다음 단계에서 찾을 gap의 후보입니다.

#### 3. 어디서 gap을 찾나 — 3가지 유형

**화면**
- **모순/충돌** — 논문 A와 B의 주장이 다름 (예: 6h vs 7–9h)
- **미검증** — 주장은 있는데 근거·재현이 약함
- **빠진 관점** — 특정 집단·조건·조합이 안 다뤄짐

**대본/진행**

gap은 크게 세 종류로 찾으면 됩니다. 첫째 모순·충돌 — 논문 A와 B의 주장이 다른 경우, 예를 들어 권장 수면이 6시간이다 vs 7~9시간이다처럼요. 둘째 미검증 — 주장은 있는데 근거나 재현이 약한 경우. 셋째 빠진 관점 — 특정 집단이나 조건, 조합이 아예 안 다뤄진 경우입니다. 어떤 유형이든 '어느 논문에서 나왔는지' 근거를 꼭 표기하게 하세요. 근거 없는 gap은 가설로 이어지면 환각이 됩니다.

#### 4. 자료 파악 & gap 발견 실습

**화면**
- 단계: papers/ 전체 구조화 추출 → 합의 vs 논쟁 지도 작성 → gap 후보 추출(유형별) → 각 gap에 근거 표기
- 산출물: 자료 지도 + gap 목록 (근거 포함) 규칙: examples/research-agent/CLAUDE.md

**대본/진행**

*실습·진행 슬라이드 — 강사가 직접 진행 (대본 생략)*

### 오후 특강 2 · 팀별 리서치 AI Agent — 가설 생성 & 평가  (90분 · 14:30–16:00)

*세션 목표: gap에서 가설을 다수 생성하고, 근거·새로움·타당성으로 평가해 제안을 확정한다.*

#### 1. 좋은 가설의 4가지 기준

**화면**
- **근거·rationale** — 실제 gap·논문에 기반했나
- **새로움·중요성** — 이미 알려진 게 아니고 풀 가치 있나
- **타당성·feasibility** — 그럴듯한가 · 검증 방법이 있나

**대본/진행**

가설을 만들기 전에, 무엇이 '좋은 가설'인지 기준을 먼저 세웁니다. 넷이에요 — 근거(실제 gap·논문에 기반했나), 새로움(이미 알려진 게 아닌가), 중요성(풀 가치가 있나), 타당성(그럴듯하고 검증할 방법이 있나). 이 기준이 곧 나중에 무엇을 제안할지 고르는 잣대가 됩니다. 만들면서도 이걸 의식하면 질이 올라가요.

#### 2. 가설 생성 방법론 — MOOSE-Chem 청사진

**화면**
- 영감 검색(논문에서 아이디어) → 조합 생성(배경+영감 → 가설) → 순위(평가·랭킹)

**대본/진행**

가설을 막 만들기 전에, 연구로 검증된 방법 하나를 청사진으로 빌려 옵니다. MOOSE-Chem이라는 연구인데(ICLR 2025, 원래는 화학용), 가설 생성을 세 단계로 봅니다 — 먼저 논문들에서 '영감', 즉 쓸 만한 아이디어를 검색하고, 그 영감을 배경 질문과 조합해 가설을 만들고, 마지막에 순위를 매깁니다. 우리 흐름과 거의 똑같죠 — 영감 검색은 자료 파악·gap, 조합은 가설 생성, 순위는 평가에 해당합니다. 그대로 돌리진 않고(설치·비용 문제) 이 3단계 아이디어만 Claude Code로 간소화해 차용합니다.

#### 3. 가설 만들기 — 두 갈래

**화면**
- gap 기반: 모순·미검증·공백에서 · '~조건에서 ~할 것이다' · 근거: 어느 gap
- 조합(영감) 기반: 다른 논문 아이디어를 끌어와 · 배경 + 영감 = 새 가설 · 근거: 어느 논문

**대본/진행**

그래서 가설을 두 갈래로 만듭니다. 첫째 gap 기반 — 어제부터 봐온 방식으로, 모순·미검증·공백에서 '~한 조건에서 ~할 것이다'를 만들고 어느 gap에서 나왔는지 답니다. 둘째 조합 기반 — 방금 본 MOOSE-Chem의 핵심인데, 다른 논문의 아이디어(영감)를 끌어와 배경과 조합해 새 가설을 만듭니다. 예를 들어 A 논문의 방법을 B 주제에 적용하면? 같은 식이죠. 이 두 갈래를 같이 쓰면 가설이 훨씬 다양해집니다. 어느 쪽이든 근거(어느 gap·어느 논문)를 반드시 답니다.

#### 4. 많이·다양하게 — 그리고 제약

**화면**
- **각 갈래에서 여러 개** — 하나가 아니라 6~10개 — breadth
- **검증 가능·근거 태그** — 한 문장 + 어디서 나왔는지
- **이미 있는 건 제외** — 가용 데이터/도구로 제약
- **예시** — '단시간 수면자는 특정 유전형에서 인지저하가 덜할 것이다'

**대본/진행**

두 갈래 각각에서 하나만 만들지 말고 여러 개 — 합쳐서 6개에서 10개씩 만드세요. 어제 토론한 '다수 생성' 전략입니다. 단 세 가지 제약 — 검증 가능한 한 문장일 것, 어디서 나왔는지 근거를 달 것, 그리고 우리가 가진 데이터·도구로 확인 가능한 범위로 좁힐 것. 이미 잘 알려진 건 빼고요. 예를 들면 '단시간 수면자는 특정 유전형에서 인지 저하가 덜할 것이다' 같은 식입니다. 많이·다양하게가 이 단계의 목표예요.

#### 5. 생성 → 선별 (강의 쟁점 적용)

**화면**
- 생성 (breadth): 많이·다양하게 · 데이터/도구로 제약 · 이미 있는 건 제외
- 선별 (우선순위): 근거·새로움·중요성 · 검증 가능성 · 상위 1~2개 추리기

**대본/진행**

여기가 어제 배운 '생성 vs 선별'을 직접 쓰는 자리입니다. 왼쪽이 생성 — 많이, 다양하게, 단 우리가 가진 데이터로 확인 가능한 범위로 제약하고 이미 있는 건 빼요. 오른쪽이 선별 — 만든 것들을 근거·새로움·중요성·검증 가능성으로 평가해서 상위 한두 개로 추립니다. 핵심은, 미리 똑똑하게 하나만 고르려 하지 말고 '많이 만들고 기준으로 추리는' 순서로 가는 겁니다.

#### 6. 제안 전 마지막 점검

**화면**
- **근거 있나?** — 막연한 상상 아닌, gap·출처 기반
- **새로운가?** — 기존 논문에 이미 있는 건 아닌지
- **검증 아이디어 있나?** — 어떻게 확인할지 한 줄이라도

**대본/진행**

제안을 확정하기 전 세 가지를 마지막으로 점검하세요. 근거가 있나 — 막연한 상상이 아니라 gap과 출처에 기반했나. 새로운가 — 기존 논문에 이미 나온 건 아닌지. 검증 아이디어가 있나 — 어떻게 확인할지 한 줄이라도 적혀 있나. 셋 중 하나라도 약하면 그 가설은 되돌아가서 다시 만들거나 보강합니다. 이 반복이 좋은 제안을 만듭니다.

#### 7. 가설 생성·평가 → 제안 확정

**화면**
- 단계: gap에서 가설 다수 생성 → 기준별 평가·점수·랭킹 → 우선순위 1~2개 + rationale 보강 → 가설 제안 확정

### 오후 특강 1 · 가설 제안서 및 발표자료 — AI Agent 이론 및 수행  (80분 · 13:00–14:20)

*세션 목표: AI 에이전트를 활용해 가설 제안서와 발표자료를 빠르게 만들고 다듬는다.*

#### 1. AI 에이전트로 산출물 만들기

**화면**
- **무엇을** — 가설 제안서·발표 슬라이드
- **왜** — 빠른 초안·반복 수정 — 시간 절약
- **원칙** — 초안은 AI, 판단·검증은 사람

**대본/진행**

이제 만든 가설 제안을 잘 '전달'할 차례입니다. 제안서와 발표 슬라이드를, 우리가 배운 AI 에이전트로 직접 만들어요. 장점은 빠른 초안과 손쉬운 수정. 원칙은 하나 — 초안은 AI가, 판단·검증은 사람이.

#### 2. 좋은 (가설)제안·발표의 핵심

**화면**
- **하나의 핵심 가설(메시지)** — 무엇을 제안하는가
- **구조: 문제(gap)→가설→근거(rationale)→의미** — 스토리
- **근거·출처로 뒷받침** — 신뢰

**대본/진행**

좋은 제안의 핵심은 셋이에요. 하나의 핵심 가설 — 청중이 가져갈 단 하나의 제안. 문제(gap)→가설→근거(rationale)→의미로 이어지는 스토리. 그리고 근거·출처로 뒷받침하는 것. 이게 분명해야 AI에게 제대로 시킬 수 있습니다.

#### 3. AI로 자료 만드는 흐름

**화면**
- 자료·근거 정리(가설·rationale 입력) → 초안 생성(제안서/슬라이드) → 검토·수정(사실·톤·구조) → 완성(최종 다듬기)

**대본/진행**

흐름은 네 단계 — 가설과 rationale·근거를 정리해 AI에게 넘기고, 제안서나 슬라이드 초안을 생성하게 한 뒤, 사실·톤·구조를 검토·수정하고, 다듬어 완성합니다. 사람이 방향을 주고 AI가 초안을 빠르게 만드는 역할 분담이 핵심이에요.

#### 4. 효과적인 프롬프트 — 보고서/슬라이드

**화면**
- **맥락 제공** — 주제·청중·목적
- **형식 지정** — 분량·구조·톤
- **자료 첨부** — 결과·근거를 함께 전달

**대본/진행**

프롬프트 요령 셋 — 맥락(분야·청중·목적), 형식(분량·구조·톤), 자료 첨부(가설·근거). 예: '발표 슬라이드 5장, 비전공자 대상, gap→가설→근거 순서로, 이 분석 노트를 근거로'처럼요. 구체적일수록 고쳐 쓰는 횟수가 줍니다.

#### 5. AI 산출물 검증·다듬기

**화면**
- **사실·출처 확인** — 환각 점검 — 필수
- **논리·구조 점검** — 메시지가 분명한가
- **톤·표현 다듬기** — 팀의 목소리로

**대본/진행**

AI 초안은 절대 그대로 내면 안 됩니다 — 반드시 검증하세요. 사실과 출처를 확인하고(특히 가설 근거가 진짜 자료에 있는지), 논리·구조가 제안을 잘 전달하는지 보고, 톤을 팀 목소리로 다듬습니다. AI 초안은 시작점일 뿐, 최종 제안은 사람이 책임집니다.

#### 6. AI Agent로 가설 제안서·발표자료 제작

**화면**
